很多静态站点的问题，不是出在复杂架构上，而是出在那些看似普通的小动作上：没有先检查工作区就提交、构建失败了还继续推、混着无关改动一起发、依赖外部接口却没考虑网络异常、改了内容却忘了确认生成结果。

这些问题单次出现时未必严重，但长期维护下来，会不断消耗耐心。因此比起追求“最自动化”的流程，我更看重“最稳的习惯”。

这篇文章记录几个我认为很值得长期保留的小习惯。

## 一、在提交前先看一眼工作区

最简单也最容易被跳过的一步，就是：

```bash
make git-status
make git-diff
```

很多混乱提交其实不是因为不会 Git，而是因为太快。尤其是内容站点，常见情况包括：

- 改了文章又顺手改了配置。
- 构建后生成了不打算提交的文件。
- 目录里还有上一次遗留的临时修改。

先看状态，再提交，会显著减少“我怎么把这个也推上去了”的情况。

## 二、把检查和构建当成发布门槛，而不是可选项

对于静态站点来说，发布前最少应该做两件事：

```bash
make check
make build
```

`check` 负责确认内容和类型层面没有明显问题，`build` 负责确认生产环境可真正生成页面。很多内容型站点本地开发看起来正常，但到了构建阶段才会暴露：

- frontmatter 字段不合法
- 某些路由依赖缺失数据
- Markdown 或 MDX 内容触发异常
- 外部接口在构建时不可用

这些问题如果不提前拦住，就会在部署之后才被看到。

## 三、提交信息要能解释“这次为什么值得记录”

很多仓库的提交信息写成：

- `update`
- `fix`
- `change`

这类提交信息短期内看似省事，长期却非常难维护。更好的写法是带上动作和对象，例如：

- `docs: replace sample posts with pinned essays`
- `build: add make targets for blog operations`
- `content: add weekly review and deployment notes`

这样以后回看历史时，能立刻知道每次提交到底做了什么。

## 四、把危险动作做成显式确认

删除文章、清空目录、批量替换内容，这些动作都不应该做成“默认就执行”的命令。当前仓库里的 `post-delete` 和 `post-clean` 都要求显式带 `force=1`，这就是一种很实用的约束。

它的价值不在于“提高门槛”，而在于帮你多停一秒钟，确认自己真的要做这件事。

## 五、接受构建里的外部依赖不是完全可靠的

对于这个站点来说，`anime` 页面会在构建时尝试访问外部数据源。这意味着构建结果不完全只受本地代码影响，还会受到网络条件影响。

这类情况不一定能立刻彻底消除，但至少要在维护习惯里承认它的存在：

- 构建失败时先区分是本地改动问题还是远程请求问题。
- 对网络依赖保留容错提示。
- 不把外部服务的瞬时失败误判为内容系统本身损坏。

只要这个边界足够清楚，排查效率就会高很多。

## 六、尽量让发布动作可重复

一个好的发布流程，最好是：

- 今天这样做能成功。
- 三周后照样做也能成功。
- 换一个心态不那么好的晚上，仍然能完成。

这就是为什么我更偏好 `Makefile` 这种固定入口。它不是最花哨的方案，但它非常适合把“记忆中的流程”变成“仓库里的流程”。

## 七、适合长期维护的最小发布流程

如果只保留最核心的动作，我认为下面这套顺序已经够用了：

1. `make git-status`
2. `make check`
3. `make build`
4. `make git-add`
5. `make git-commit msg="..."`
6. `make git-push`

如果工作区比较干净，也可以直接走：

```bash
make publish msg="content: add new blog posts"
```

但前提仍然是不把无关变更混进来。

## 八、真正让发布稳定的不是工具，而是纪律

同样的工具，在不同人手里会得到完全不同的结果。真正让静态站点发布变稳的，不是某个神奇脚本，而是一些被持续执行的小纪律：

- 先看状态，再提交。
- 先构建，再推送。
- 重要动作需要显式确认。
- 提交信息必须可读。
- 外部依赖问题和本地问题要分开判断。

这些习惯看起来普通，但它们决定了这个博客未来是不是能长期低成本维护。

对于一个内容站点来说，稳定发布本身就是内容生产力的一部分。
